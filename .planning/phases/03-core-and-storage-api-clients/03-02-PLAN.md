## Key Infrastructure from Phase 2

**_BaseClient interface:**
- `_sync_request(method, url, **kwargs)` — Synchronously makes HTTP request with auth injection and error mapping
- `api_key` property — This will be the storage zone password for StorageClient; injected as `AccessKey` header

**Response format:**
- All methods return plain `dict` or `bytes` from response
- `upload()` returns response dict on success
- `download()` returns bytes (response.content)
- `delete()` returns nothing (or empty dict)
- `list()` returns list[dict] from response.json()

## Authentication Mechanism

StorageClient uses HTTP Basic Auth, not the AccessKey header. Pattern:
- Encode zone_name:password in base64
- Inject as Authorization: Basic <base64> header
- Bunny Storage API validates credentials per zone

Note: _BaseClient injects AccessKey header automatically. For StorageClient, we override by passing explicit Authorization header in kwargs to _sync_request (this supplements/overrides the AccessKey header).

## File Path Handling

All paths passed to Storage API endpoints should be URL-encoded by httpx automatically. Caller provides path (e.g., "/photos/2024/image.jpg"). httpx.request(url=..., content=...) handles encoding.

For list endpoint, path should end with "/" to list directory (e.g., list("photos/") lists photos/ directory).

- Create REGION_MAP (dict or class attribute) mapping region names to base URLs per STOR-05 specification
- Determine base_url by looking up region in REGION_MAP; default "falkenstein" → "https://storage.bunnycdn.com"

**Helper method (_build_auth_header):**
- Take zone_name and password, encode as base64: `import base64; base64.b64encode(f"{zone_name}:{password}".encode()).decode()`
- Return dict: `{"Authorization": f"Basic {encoded}"}`

**upload() method (STOR-01):**
- Signature: `upload(self, path: str, data: bytes | BinaryIO, content_type: str | None = None) -> dict`
- Build URL: `{base_url}/{zone_name}/{path}` (httpx will URL-encode path)
- If data is BinaryIO, read it or pass directly to httpx (httpx can handle file-like objects)
- HTTP method: PUT
- Headers: Include auth header from _build_auth_header(); if content_type is provided, add Content-Type header
- Return: response.json() (dict)

**download() method (STOR-02):**
- Signature: `download(self, path: str) -> bytes`
- Build URL: `{base_url}/{zone_name}/{path}`
- HTTP method: GET
- Headers: Auth header
- Return: response.content (bytes)

**delete() method (STOR-03):**
- Signature: `delete(self, path: str) -> None`
- Build URL: `{base_url}/{zone_name}/{path}`
- HTTP method: DELETE
- Headers: Auth header
- Return: None (do not return response)

**list() method (STOR-04):**
- Signature: `list(self, path: str = "/") -> list[dict]`
- Build URL: `{base_url}/{zone_name}/{path}` (note: path should end with "/" to list directory)
- HTTP method: GET
- Headers: Auth header
- Return: response.json() (should be list[dict] of files/folders)

**Implementation notes:**
- All requests use _sync_request(method, url, headers=..., content=... if upload)
- For upload with BinaryIO, pass data directly to httpx; httpx reads it
- For download, use response.content (not response.json())
- All auth headers bypass AccessKey injection from _BaseClient by explicitly passing Authorization header
- Region lookup: case-sensitive matching against REGION_MAP; raise ValueError if region not found
- Zone and password are part of path/auth; no additional validation needed

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User code → StorageClient | User supplies zone_name, password, region; credentials kept in memory |
| StorageClient → Bunny Storage API | Credentials encoded in Authorization header (Basic Auth); transmitted over HTTPS |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Spoofing | zone_name and password parameters | mitigate | User supplies credentials; SDK encodes in Basic Auth header per HTTPS spec. No validation of credentials format; Bunny API validates. |
| T-03-08 | Information Disclosure | Password stored in StorageClient.password attribute | accept | Password must be stored to construct auth header for each request. Users should not log or share client instances. |
| T-03-09 | Denial of Service | Large file uploads via upload() with BinaryIO | mitigate | httpx handles streaming; SDK does not buffer entire file in memory. Users should handle timeouts via httpx timeout settings. |
| T-03-10 | Elevation of Privilege | Zone access via password | mitigate | Zone password is secret; SDK transmits via HTTPS Basic Auth. Bunny API enforces zone ownership validation. |
| T-03-11 | Tampering | HTTP MITM on Storage API requests | mitigate | Use HTTPS URLs (hardcoded base_url patterns all use https://). httpx validates TLS certificates by default. |
| T-03-12 | Repudiation | Audit trail of file operations | accept | SDK does not log operations. Bunny Storage API provides audit trail. User can wrap client methods with logging. |
</threat_model>
