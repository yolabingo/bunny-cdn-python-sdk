    ```python
    """Exception hierarchy for bunny-cdn-sdk."""

    from __future__ import annotations

    from typing import TYPE_CHECKING

    if TYPE_CHECKING:
        import httpx

    __all__ = [
        "BunnySDKError",
        "BunnyAPIError",
        "BunnyAuthenticationError",
        "BunnyNotFoundError",
        "BunnyRateLimitError",
        "BunnyServerError",
        "BunnyConnectionError",
        "BunnyTimeoutError",
    ]


    class BunnySDKError(Exception):
        """Base class for all bunny-cdn-sdk errors."""


    class BunnyAPIError(BunnySDKError):
        """Raised when the Bunny CDN API returns an HTTP error response."""

        def __init__(
            self,
            status_code: int,
            message: str,
            response: httpx.Response,
        ) -> None:
            super().__init__(f"HTTP {status_code}: {message}")
            self.status_code = status_code
            self.message = message
            self.response = response

        def __str__(self) -> str:
            return f"HTTP {self.status_code}: {self.message}"


    class BunnyAuthenticationError(BunnyAPIError):
        """Raised on HTTP 401 Unauthorized responses."""


    class BunnyNotFoundError(BunnyAPIError):
        """Raised on HTTP 404 Not Found responses."""


    class BunnyRateLimitError(BunnyAPIError):
        """Raised on HTTP 429 Too Many Requests responses."""


    class BunnyServerError(BunnyAPIError):
        """Raised on HTTP 5xx Server Error responses."""


    class BunnyConnectionError(BunnySDKError):
        """Raised when a network-level connection failure occurs (not an HTTP response)."""


    class BunnyTimeoutError(BunnyConnectionError):
        """Raised when a request times out before receiving a response."""
    ```

    Notes on this implementation:
    - `from __future__ import annotations` enables PEP 563 deferred evaluation, allowing `httpx.Response` in TYPE_CHECKING block to work as a forward reference at runtime without importing httpx at module load
    - `__all__` satisfies ruff's implicit-namespace-packages and re-export rules
    - `BunnyConnectionError` and `BunnyTimeoutError` inherit only from `BunnySDKError` / `BunnyConnectionError` respectively — they have NO `status_code`, `message`, or `response` attributes (per D-04)
    - `__str__` is explicit (not relying on Exception.__str__) for predictable output
    - All classes have docstrings to satisfy any D-class rules that may be active (even though D is globally ignored, explicit docstrings are cleaner)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Exception message → user/logs | status_code and message from API responses are surfaced in exception __str__; attacker-controlled response bodies could inject content |
| httpx.Response reference | Storing the raw response object in the exception means response body stays in memory as long as the exception is alive |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | BunnyAPIError.__str__ | accept | message field comes from Bunny CDN API responses; the SDK does not sanitize it — callers choose whether/how to log. Document in Phase 4 that exception messages may contain API-sourced strings. |
| T-02-02 | Information Disclosure | BunnyAPIError.response | mitigate | response stores the full httpx.Response including headers (which may contain partial auth tokens in error responses from misconfigured proxies). Phase 2 _client.py should NOT pass through raw AccessKey headers in error response objects — acceptable because httpx.Response is the server's response, not the outbound request. No action needed in _exceptions.py itself. |
| T-02-03 | Denial of Service | Exception memory retention | accept | Large API error response bodies held in BunnyAPIError.response. Low risk: SDK is caller-controlled; callers who need memory safety can del exc or use finally blocks. No SDK-level mitigation warranted. |
| T-02-04 | Spoofing | BunnyAPIError subclass selection (401/404/429/5xx) | mitigate | Phase 2 _client.py must map HTTP status codes to exception types using an exhaustive match/if-elif chain with explicit fallback to BunnyAPIError for unmapped 4xx. Misclassification would give callers false signal about error type. Enforce in Phase 2 plan. |
</threat_model>
